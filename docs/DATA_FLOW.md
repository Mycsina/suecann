# Data Flow Architecture — Sueca WANN

> Auto-generated by optimization-auditor skill. Last audited: 2026-06-02.

## Module: WANN Forward Pass (`wann_network.rs`)

- **Input:** Belief state (30× f64) + shared weight (f64) + pre-allocated scratchpad
- **Output:** 4 intent activation values written into scratchpad[22..26]
- **Hot path:** `RustWannNetwork::forward()` — called **~20M times per generation**, ~24B times across a full 1200-gen run (1000 genomes × 512 games × 40 decisions × 1 weight)
- **I/O boundaries:** None — zero heap allocations in the hot path. All data lives on the stack/scratchpad.
- **Memory profile:** Scratchpad pre-allocated per thread (size = num_nodes × 8 bytes). CSR arrays read-only after construction.
- **Key optimizations already applied:**
  - CSR format for cache-locality on connection lookups
  - Aggregation-type branch hoisted outside the inner edge loop (avoids per-connection branch mispredicts)
  - Sign inversion computed inline (`1.0 - val`) rather than stored separately
  - Identical sign + weight computation unrolled across all three aggregation branches
- **Bottleneck potential:** The sign branch inside the edge loop (`if sign == -1`) is still per-connection. For networks with hundreds of connections, branch predictor should handle this well since signs are random and unpredictable per connection.
- **CUDA candidacy:** **Medium** — batch inference (same network, many belief states) is the natural fit. However, topology varies per genome so batched ragged-CSR inference across different genomes would need custom kernels.

## Module: Genome Evaluation (`evaluator.rs`)

- **Input:** Genome + deal set + opponent pool (HOF + MAP-Elites)
- **Output:** Delta-fitness (game points vs HeuristicBot baseline)
- **Hot path:** `evaluate_genome_delta()` — loops over 128 deals × 4 seat rotations = 512 games per genome
- **I/O boundaries:**
  - **RNG calls:** Per-game deal generation (seeded, deterministic). LCG — cheap but called frequently.
  - **HOF/MAP-Elites sampling:** Read-only lookups into pre-built archives.
- **Memory profile:** Game state is stack-allocated. Cached deal worlds stored in `Vec<CachedWorld>`. No per-evaluation allocation.
- **Parallelism:** Rayon `par_iter()` over genome population. Each thread gets its own scratchpad and RNG.
- **Weight sweep loop:** Currently single-weight `[1.0]`, but designed for multi-weight sweep. Each additional weight doubles the forward() call count in the hot path.
- **CUDA candidacy:** **High** — the 512 games per genome are completely independent. Each game decision could batch its forward() call. OR the entire population of 1000 genomes could be evaluated in parallel GPU-side if the game simulation was ported.

## Module: Game Simulation (`simulator.rs`, `engine.rs`)

- **Input:** Deal + seat assignment + bot configurations
- **Output:** Game result (trick outcomes, point totals)
- **Hot path:** Inner trick loop — ~10 tricks × 4 card plays = ~40 decisions per game
- **I/O boundaries:** Only RNG calls for deal shuffling. Bitboard operations are all CPU register-level.
- **Memory profile:** Bitboard game state (small, copyable). Belief encoder reads game state bitboards.
- **CUDA candidacy:** **Low** — bitboard operations don't map well to GPU. The game simulation is already lightweight; the bottleneck is the WANN inference inside it, not the simulation itself.

## Module: Checkpoint I/O (`checkpoint.rs`)

- **Input:** TrainingState struct (population snapshots, HOF, generation counter)
- **Output:** `training_state.bin` (bincode-serialized binary)
- **I/O boundaries:** File write every 10 generations (configurable). `BufWriter` + `bincode::serialize_into` — already buffered and binary.
- **Memory profile:** Full population serialization — 1000 genomes × (nodes + connections). Estimated ~10-50MB per checkpoint. Serialized in-memory before write.
- **Bottleneck potential:** **Low** — infrequent (every 10 gens). Bincode is fast. BufWriter prevents syscall overhead.
- **CUDA candidacy:** **None** — disk I/O, not compute-bound.

## Module: Dataset Loading (`dataset.rs`)

- **Input:** `.npz` file (zipped NumPy archive)
- **Output:** `ExpertDataset` with states, intents, legal masks, lead flags
- **I/O boundaries:** File read once at startup. `BufReader` + `ZipArchive` + `NpyFile`. Already buffered.
- **Memory profile:** Full dataset loaded into memory (~100K states × (30 × 4 + 4 + 40) bytes ≈ ~20MB). One-time cost.
- **CUDA candidacy:** **None** — one-time load, not a hot path.

## Module: CSV Stats Output (`train.rs`)

- **Input:** Per-generation metrics (fitness, species count, connection count)
- **Output:** `training_stats.csv` (one line per generation, ~150 bytes)
- **I/O boundaries:** Unbuffered `File` write + explicit `flush()` every generation. Total ~180KB over a full run.
- **Bottleneck potential:** **Negligible** — 1200 writes over 3-4 hours. The unbuffered write is not measurable against the compute cost.
- **CUDA candidacy:** **None** — trivial I/O.
